Distinguishing BGA table types (tournament / arena / regular). Verified by dumping `gameui` and the `tableinfos` API on live tables of each type (May 2026).

**Tournament** — detectable from `gameui` for free (no extra call):
- `gameui.tournament_id` is a positive number on tournament tables; `null` otherwise.
- `gameui.tournament_info` = `{id, name, championship_name, avatar}` when in a tournament.
- `tableinfos.has_tournament === "1"` is an equivalent signal.

**Arena vs regular** — the reliable signal is BGA framework **option 201 ("Game mode")**: its selected `value` is `"0"` Normal, `"1"` Friendly, `"2"` Arena (string or number; compare `String(value) === "2"`). Read from `tableinfos.options["201"].value`. NOT distinguishable from `gameui` (arena and regular tables produce byte-for-byte identical `gameui`/`gamedatas` dumps), so the table-info API `/table/table/tableinfos.html?id=<tableId>` is still required. **Do NOT use `table_matchmaking`** — it only means "matchmade", which is also true for "Play now" quick games (regular, not arena), so it over-matched (confirmed wrong on table 858993521, a matchmade real-time The Gang game with option 201 = "0" that got mislabeled arena; fixed May 2026 by switching the probe to option 201). Two approaches to reaching the API FAILED:
1. **In-page from the injected extract.ts (MAIN world):** calling `gameui.ajaxcall("/table/table/tableinfos.html", ...)` inside `extractGameData`'s promise breaks the whole extraction → `executeScript` returns no result → "Extraction failed: No result from extraction" (happened with and without `jsonp:1`, even fully guarded). Works in the devtools console (not under the same constraints), so it passed manual probing but broke the real extension.
2. **Plain `fetch()` from the background service worker** (`credentials:"include"` + `x-requested-with`): BGA REJECTS it — the SW request can't carry BGA's session **request token**, so the response is BGA's error envelope `{status, exception, error, expected, code, args}` (confirmed: `players: 0`). Do not use.

**Working approach:** a SEPARATE page injection, decoupled from extraction. `background.ts` injects `probeTableTypeFn` via `chrome.scripting.executeScript({ func, world: "MAIN" })` (its own call, NOT part of extractGameData). It calls `gui.ajaxcall("/table/table/tableinfos.html", { id }, gui, ok, err)` (no jsonp) — `gameui.ajaxcall` adds the session token the raw fetch lacked. Self-contained, times out to null, so a failure only loses the type (extraction unaffected). Gated by an in-memory `probedTableTypes` Set so it runs once per table per SW lifetime.
- **Arena ⟺ `options["201"].value` (Game mode) === "2"**. Tournament flagged by `has_tournament === "1"`, and by `gameui.tournament_id > 0` (which extract.ts uses directly — no API call needed). (Historical: the probe used to read `table_matchmaking === "1"` off players — wrong, see above. The `players` objects also differ in shape by table status: live tables carry `table_matchmaking`/`arena_points`/`league_number`; FINISHED tables drop those and instead hold `{awards:[...]}` plus a top-level `result` block — another reason not to depend on player fields. `options` is present in both states.)
- `setTableType` is **first-write-wins** (classify once, remember — don't re-probe every open). The probe is skipped entirely when `isTableTypeKnown(tableId)` is already true. All three types are stored (regular too, so regular tables aren't re-probed).
- Real-time mode (stopwatch, `bgaa_time_modes`) is a SEPARATE orthogonal flag — a real-time table can be regular/arena/tournament. Don't conflate or clear mode flags when touching types.
- A broken interim build (SW fetch) mis-stored arena tables as "regular". This was fixed with a one-time guarded migration that dropped non-tournament type entries so they re-classified via the corrected probe; that migration code has since been removed (the bad data is gone). Prefer a one-time cleanup over switching to overwrite semantics — overwrite re-classifies on every open and loses the first-write-wins benefit.
- Icons (in `sidepanel.ts`, both Sessions + Tables views, unified `tableCell`): real-time = stopwatch (`#7db8f0`), tournament = trophy (`#d4b24a`), arena = **filled 3D colosseum** (`ARENA_SVG`, `#d99a5b`) — solid cylinder silhouette with a hollow oval top opening + arched window cutouts (single `<path fill-rule="evenodd">`), matching a user reference. All three are filled/solid for consistency. Rejected en route: crossed-swords, a flat colosseum facade, and a stroke/outline stadium bowl (user wanted solid with empty top).
- Persistence model: real-time mode (`bgaa_time_modes`) and table type (`bgaa_time_types`) are per-table maps in `chrome.storage.local`, populated ONLY when a table is extracted/probed with the side panel open — classification needs the live game page, so session history (focus-tracked, automatic) can't be retroactively classified. Both survive reloads. CSV columns: `game,game_id,table_id,from,to,minutes,realtime,type` (game=display name, game_id=URL slug). Import parses by HEADER NAME (robust to reordered columns; game/game_id/table_id/from/to required, realtime/type optional — a CSV without game_id imports nothing, no name-as-slug fallback), restoring the session slug, the slug→name map, modes, and types (first-value-wins). Export→import is now LOSSLESS. The BGA-page localStorage backup/restore (`backupToBga`/`restoreFromBga`) REUSES exportSessionsCsv/importSessionsCsv (stores the CSV string), so reinstall restores everything too. (Legacy-JSON-blob restore support was removed.)
- Icon-design workflow that worked well: generate candidate SVG path strings (Node helper for arches/ellipses, round to 2 decimals), drop them into a throwaway HTML at sizes 11/16/24/48px on the panel's dark bg, render with `chrome --headless=new --screenshot --window-size=W,H file://...` (Chrome at `C:\Program Files\Google\Chrome\Application\chrome.exe`), and Read the PNG to eyeball legibility at the real 11px size. Reliable without a live browser (chrome-devtools MCP needs a debug port that wasn't running). Clean up the temp `_icontest.html`/`_iconshot.png` after.

**Red herrings (do NOT use):**
- `league_number` — present and EQUAL (e.g. "6") for both arena and regular. Not a discriminator.
- `arena_points` — present on players regardless of table type.
- `level_filter.details` labels DO differ (arena = league tiers Bronze/Silver/Gold/Diamond/Platinum/Elite; non-arena = skill levels Beginners/Apprentices/.../Masters) but are localized → fragile, avoid string-matching.

**Resolved caveat (May 2026):** the old `table_matchmaking === "1"` heuristic assumed matchmaking ⟺ arena. That was WRONG — "Play now" quick games are matchmade but regular. Observed on table 858993521 (matchmade real-time The Gang, mislabeled arena). Now uses `options["201"].value === "2"` instead. Regression test: `probeTableTypeFn` suite in `background.test.ts` (the probe is exported solely so the test can drive it with faked `gameui`/`window`, matching the `watcherFunction` precedent).

**In the extension:** `extract.ts` fetches `tableinfos` in parallel with notificationHistory and derives `RawExtractionData.tableType` (`"tournament" | "arena" | "regular" | null`). Tournament from `gameui.tournament_id`; arena/regular from `table_matchmaking`; `null` when tableinfos is unavailable and it's not a tournament (so we don't mislabel). Stored per-table in `chrome.storage.local` under `bgaa_time_types` (first-write-wins, fixed per table), orthogonal to the real-time flag in `bgaa_time_modes`. Rendered as a trophy (tournament) / crossed-swords (arena) icon in the play-time Tables view. See [[project_bga_mode_detection]].
